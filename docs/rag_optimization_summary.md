รายงานฉบับนี้สรุปคำตอบของคำถามในแบบฝึกหัด (Workshop Exercise) และผลลัพธ์การปรับแต่งระบบสืบค้นข้อมูลร่วมกับตัวสร้างคำตอบ (Retrieval-Augmented Generation หรือ RAG) ของแชตบอตเอกสาร Blendata Enterprise v4.6.0

## 1. คำตอบสำหรับแบบฝึกหัด (Step 1 — Profiling Answers)

### **คำถาม: ปัญหาความล่าช้า (Slowdown) เกิดขึ้นที่ขั้นตอนใด? ระหว่าง "Retrieve" หรือ "Generate" และเป็นเพราะอะไร?**

* **คำตอบ**: ปัญหาความล่าช้าเกิดขึ้นที่ขั้นตอน **"Generate" (LLM Generation)**
* **วิเคราะห์สาเหตุเชิงลึก**:
  1. **ขนาดของบริบท (Context Too Large)**: ค่าเริ่มต้นกำหนดให้ดึงข้อมูลจากฐานข้อมูลถึง 20 ชิ้น (`RETRIEVAL_K = 20`) ส่งผลให้ Prompt มีขนาดใหญ่ถึงประมาณ ~2,100 tokens ส่งไปยัง API ของ Groq ทำให้ใช้เวลาประมวลผลสูงมาก
  2. **ไม่มีการจำกัดผลลัพธ์คำตอบ (`MAX_TOKENS = None`)**: ทำให้ LLM เสียเวลาไปกับการสร้างคำตอบที่ยาวเกินความจำเป็น ส่งผลให้ความเร็วเฉลี่ยตกไปอยู่ที่ **11.9 วินาทีต่อคำถาม**
  3. **การดึงข้อมูลทำงานเร็วปกติ (Retrieval is Fast)**: ในฝั่งฐานข้อมูล Qdrant ใช้เวลาดึงข้อมูลเฉลี่ยเพียง **24ms - 25ms** เท่านั้น

## 3. สรุปวิธีการปรับปรุงระบบ (Applied Optimization Techniques)

เราแบ่งการแก้ไขออกเป็น 3 ขั้นตอนย่อยในไฟล์ระบบหลักของแชตบอต ดังนี้:

### **Step 2A — การปรับตั้งค่าระบบและการรันแบบขนาน (Config & Async)**
* **ปรับพารามิเตอร์ของ RAG**:
  * ลด `RETRIEVAL_K = 5` (จากเดิม 20)
  * ตั้งค่าคะแนนความถูกต้องขั้นต่ำ `SCORE_THRESHOLD = 0.35` เพื่อกรองเนื้อหาไม่เกี่ยวข้องทิ้ง
  * จำกัดคำตอบสูงสุด `MAX_TOKENS = 512` และจำกัดความจำประวัติแชต `MAX_HISTORY = 10`
* **เปลี่ยนโหนดตอบคำถามเป็น Asynchronous (`exercise_generate_node`)**:
  * เปลี่ยนหัวฟังก์ชันเป็น `async def exercise_generate_node`
  * เปลี่ยนการสร้างคลาสเชื่อมต่อ API Groq มาเรียกใช้ผ่านรูปแบบ Singleton client `_get_llm()` เพื่อไม่ต้องเชื่อมต่อใหม่ทุกครั้ง
  * ปรับแต่งคำสั่งเรียกใช้ LLM จากการบล็อกระบบ (`invoke`) ไปเป็นแบบไม่บล็อกการทำงานหลัก (`await llm.ainvoke(messages)`)

### **Step 2B — การจัดเรียงและกรองข้อมูลระดับเวกเตอร์ (Reranking & Filtering)**
* ปรับปรุงฟังก์ชัน `exercise_retrieve_node` ให้อ่านและกรองข้อมูลเอกสารที่มีประสิทธิภาพขึ้น:
  ```python
  # จัดเรียงคะแนนความเกี่ยวข้อง (Cosine Similarity) จากมากไปน้อย
  ranked = sorted(docs_with_scores, key=lambda x: x[1], reverse=True)
  
  # คัดเฉพาะชิ้นที่คะแนนผ่าน Threshold และเลือกเพียง 3 ชิ้นที่ดีที่สุด (Top 3)
  top_docs = [
      (doc, score) for doc, score in ranked
      if SCORE_THRESHOLD is None or score >= SCORE_THRESHOLD
  ][:3]
  ```

### **Step 2C — ระบบส่งคำตอบทีละอักษร (SSE Token Streaming)**
* แก้ไขฟังก์ชัน `event_generator` ในไฟล์เร้าเตอร์แชต [exercise.py](file:///Users/natthachak/Documents/SparkCamp/BlenData_June30/tu-ai-workshop/app/routers/exercise.py) ให้ส่ง Token คืนสู่ผู้ใช้ทันทีที่ได้รับข้อมูลการพิมพ์จากเซิร์ฟเวอร์โดยไม่ต้องรอให้ประโยคยาวจบสมบูรณ์:
  ```python
  async for chunk in llm.astream(messages):
      if chunk.content:
          yield f"data: {chunk.content}\n\n"
  ```

## 4. ผลการรันระบบกระจายโหลดคู่ขนาน (Nginx + Multi-Container Load Balancing)

เราได้ขยายสเกลระบบหลังบ้านของ API ขึ้นมา **3 ตัว (3 Instances)** และต่อผ่านระบบกระจายโหลด **Nginx Load Balancer** บนพอร์ต `8080` เพื่อจำลองการรองรับการเข้าใช้งานพร้อมกันปริมาณสูง (Burst Test 5 ผู้ใช้ยิงคำถามพร้อมกันทันที):

| ตัวชี้วัดภาระโหลด (Burst 5 Users) | BAD Endpoint | GOOD Endpoint (สเกลคู่ขนาน + Nginx) |
| :--- | :---: | :---: |
| **เวลารวมในการเคลียร์คิว (Wall Clock Time)** | 21.2 วินาที | **2.7 วินาที** (เร็วขึ้น 7.8 เท่า) |
| **อัตราการตอบกลับสำเร็จ (Throughput)** | 0.05 req/s | **1.84 req/s** (สูงขึ้น 36.8 เท่า) |
| **ความหน่วงเฉลี่ยของผู้ใช้ (Avg Latency)** | 21,166 ms | **864 ms** (ต่ำกว่า 1 วินาที!) |
| **ความช้าสูงสุด (P95 Latency)** | 21,166 ms | **2,714 ms** |
| **จำนวนสายที่เกิดข้อผิดพลาด (Errors)** | 4 ครั้ง (เกือบล่มทั้งหมด) | **0 ครั้ง (ผ่าน 100%)** |

### **วิเคราะห์ความสำเร็จการสเกลคู่ขนาน (Scaling Insight)**
1. **การจำกัดคิวลดน้อยลง**: เมื่อเรามี API 3 ตัวแบ่งกันรับงาน Nginx จะช่วยกระจายผู้ใช้เข้าเครื่องที่ว่างอยู่ทันที ทำให้ผู้ใช้ไม่ต้องยืนรอกันแบบคิวยาวเป็นเส้นตรง (FIFO Queue Delay) ทำให้ความหน่วงเฉลี่ยลดลงจาก 4.7 วินาที (รันแบบ Instance เดียว) เหลือเพียง **864ms**
2. **การทำ Routing ที่ชาญฉลาด**: การสเกลระบบร่วมกับการกรอง RAG ที่กระชับขึ้น ช่วยป้องกันไม่ให้การประมวลผล LLM ค้างอยู่ในหน่วยความจำตัวเดียว และทำให้อัตราความล้มเหลว (Errors) กลายเป็นศูนย์ (0 Errors) แม้จะยิงถล่มมาพร้อมกันก็ตาม
